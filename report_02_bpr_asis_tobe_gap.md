# レポート2：BPR設計の実務 ― ASIS/TOBE/GAP分析と業務再設計

---

## 1. BPRと「業務改善」の違い

| 比較軸 | 業務改善（カイゼン） | BPR（Business Process Reengineering） |
|---|---|---|
| スコープ | 個別業務・局所最適 | 業務全体・プロセス横断 |
| 変化の深さ | 漸進的（10〜30%改善） | 抜本的（50〜80%の変革） |
| 問いの立て方 | 「もっと速くできないか？」 | 「そもそもこの業務は必要か？」 |
| リスク | 低い | 高い（抵抗・失敗リスクあり） |
| 代表手法 | QCサークル、5S | プロセスマイニング、AI/RPA再設計 |

> **例え話：** 業務改善は「既存の道をきれいに舗装する」こと。BPRは「そもそも別の道を作る、または飛行機で飛ぶ」こと。太陽油脂の26名→19名（27%削減）はカイゼンでは届かない。BPRの射程距離が必要。

## 2. ASIS分析の手順

**ステップ1：業務棚卸し**
- 対象部門の全業務をリストアップ（業務名・担当者・頻度・月間工数）
- 資料だけでなく現場観察（シャドーイング）で「隠れ業務」を発掘する

**ステップ2：プロセスフロー可視化**
- 各業務の入力（インプット）→処理（アクション）→出力（アウトプット）を図示
- システム（ERP・Excel・メール）とのやり取りを明記
- 担当者をレーン（スイムレーン図）で分けると権限境界が見える

**ステップ3：ボトルネック特定**
- 工数が集中している業務（工数ヒートマップ）
- 手戻り・例外処理が多い業務（品質問題の源泉）
- 属人化が進んでいる業務（リスクの核心）

### 太陽油脂 ASIS 例（原料・包材MRP作成）

| 項目 | 現状（ASIS） |
|---|---|
| 担当者 | 生産管理担当2名 |
| 頻度 | 週1回（月曜） |
| 工数 | 約4時間/回 × 2名 = 8時間/週 |
| 手順 | ERP出力→Excelコピペ→在庫照合→発注量計算→上長確認→発注 |
| 問題 | Excelの数式ミス、在庫確認の二重作業、上長が不在だと翌日に遅延 |

## 3. TOBE設計の手順

**ステップ1：ゴール定義（定量・定性）**
- 定量：削減工数・削減人員・処理時間短縮・精度向上
- 定性：担当者が「判断業務」に集中できる状態

**ステップ2：理想プロセスフロー設計**
- 「この業務は誰がやる必要があるか？」から問い直す
- AIやRPAが担当するステップを明記（人間の役割は判断・例外処理・承認に絞る）

**ステップ3：実現条件の列挙**
- 必要なデータ（構造化・品質・取得経路）
- 必要なシステム（ERP連携・API可否・クラウド環境）
- 必要な組織変更（権限・役割・スキル再定義）

### 太陽油脂 TOBE 例（原料・包材MRP作成）

| 項目 | 将来像（TOBE） |
|---|---|
| 担当者 | RPA自動実行 + 担当者1名（例外処理・承認のみ） |
| 頻度 | 自動（毎日夜間バッチ） |
| 工数 | 確認・承認0.5時間/日 |
| 手順 | ERP→RPA自動取得→計算エンジン→結果レポート生成→担当者承認→ERP自動登録 |
| 効果 | 週8時間 → 週3.5時間（▲56%）、属人化解消、ミス率低下 |

## 4. GAP分析の3層構造

| 層 | 問い | 太陽油脂MRP例 |
|---|---|---|
| **What（何が違うか）** | ASISとTOBEの差分は何か | 手動Excelコピペが残っている／ERPからの自動データ取得がない |
| **Why（なぜ差があるか）** | 技術・組織・データの障壁は何か | ERPの外部連携ライセンスが未契約／担当者がRPA設計スキルを持たない |
| **How（どう埋めるか）** | 具体的なアクションと優先順位 | ERPライセンス拡張（3ヶ月）→PoC（1ヶ月）→本番展開（2ヶ月） |

## 5. ステークホルダー別合意設計

| ステークホルダー | 主な関心事 | 合意設計のポイント |
|---|---|---|
| **業務改革部** | プロジェクトの成果・スケジュール | 定量KPI・マイルストーンを明示。M2（5/20）に間に合う計画を示す |
| **人事部** | 人員削減の実施方法・労務リスク | 「削減」ではなく「再配置・スキルアップ」の語り方。再配置先を先に設計 |
| **親会社物流ユニット** | 自社ルール・権限への影響 | 変更前に必ず事前相談。承認フローを明確化し「勝手に変えない」を担保 |
| **現場担当者** | 自分の仕事がなくなる恐怖 | 自動化後に担当する「より価値の高い業務」を具体的に提示する |

> **合意設計の鉄則：** 反対は「最後に来る」。技術設計より先に、誰がいつ反対してくるかを予測し、先手を打つ。

## 6. 失敗パターンと回避策

| 失敗パターン | 発生場面 | 回避策 |
|---|---|---|
| ASIS調査が資料だけで終わる | 現場ヒアリング時間が確保できない | シャドーイング（半日でも）を必ず組み込む |
| TOBEが技術起点になる | ITベンダーが設計を主導する | ビジネス側がゴール定義を先に行う |
| GAPが「全部難しい」で終わる | 障壁だけ列挙して優先順位が付かない | 実現コスト×インパクトの2軸マトリクスで優先順位を付ける |
| 人事との調整が遅れる | プロジェクト後半に人員削減が具体化する | プロジェクト開始時から人事を巻き込む |
| 親会社承認が得られない | 変更後に報告する | 変更前のドラフト段階で相談・合意を取る |

## 想定質問と回答

**Q1：「ASISヒアリングで現場が本音を話してくれない場合どうするか？」**

**質問の背景：** 現場担当者は「自分の業務を詳しく話すと削減される」と感じて情報を出し渋る。顧客（業務改革部）がこの問いを出すのは、実際にヒアリングが行き詰まったか、行き詰まりを予測しているから。

**回答（推奨）：** ①ヒアリングの目的を「改善のため・効率化のため」と明示し、個人評価に使わないことを宣言する。②管理職なしで現場担当者だけと話す場を設ける。③「業務の大変なところは？」という問いより「毎週月曜の朝、最初に何をしますか？」という行動ベースの質問に変える。④実際に隣で業務を見るシャドーイングが最も有効。

他の選択肢との比較：アンケートは匿名性が保てるが深掘りができない。管理職経由のヒアリングは情報が加工される。シャドーイングは時間はかかるが最も実態に近い情報が得られる。

---

**Q2：「TOBEを描いたが予算が半分になった場合のスコープダウン基準は？」**

**質問の背景：** 役員会承認後に予算圧縮が起きるのは珍しくない。業務改革部が「どこを削るか」の判断基準を求めている。

**回答（推奨）：** スコープダウンの基準は「ROIの高い順に残す」。①削減工数インパクトが大きい業務、②PoCリスクが低い業務（ルール化可能性が高い）、③他業務への波及効果が高い業務の順で残す。太陽油脂の場合、受注入力自動化（7名→4名の3名分）は工数インパクトが最大なので最後まで残すべき対象。

他の選択肢との比較：「全体を薄く削る」スコープダウンは成果が見えにくくなる。「核心業務に集中して成果を出し、次フェーズを獲得する」戦略の方が長期的に顧客満足度が高い。

## 用語集

| 用語 | 説明 |
|---|---|
| BPR（Business Process Reengineering） | 業務プロセスを抜本的に再設計すること |
| ASIS | 現状の業務プロセス・状態 |
| TOBE | 目指すべき将来の業務プロセス・状態 |
| GAP分析 | ASISとTOBEの差異を特定し、埋める方法を設計する分析 |
| シャドーイング | 担当者の隣で業務を観察する現場調査手法 |
| スイムレーン図 | 担当者・部門を横軸に並べ、業務フローを可視化した図 |
| MRP（製造資源計画） | 製品の生産計画から原材料・部品の必要量を逆算する計画システム |
| 属人化 | 特定の人物しか業務を遂行できない状態。退職・異動でリスクが顕在化する |
| スコープダウン | プロジェクトの対象範囲を縮小すること |
