## 第1部: 重要Q&Aベスト5

### Q1. ベクトルDBに格納済みの機密情報を個別削除する方法は？

**【背景】** GDPR第17条「削除権（忘れられる権利）」や個人情報保護法第30条の利用停止請求により、ベクトルDBに埋め込まれた個人データの個別削除が必要になるケースが急増。しかし多くの企業はベクトルDBを「検索用キャッシュ」として扱い、削除プロセスを設計していない。チャンクに紐づく原文・ベクトル・メタデータを全て削除する必要がある。

**【推奨対策】** 削除キー（person_id / document_id）をメタデータに必ず付与し、削除APIを事前に実装しておく。

```python
# 例: Azure AI Searchでの削除
search_client.delete_documents(
    documents=[{"id": f"chunk_{doc_id}_{i}"} for i in chunk_indices]
)
# 原文ストレージ（Blob/S3）も同時削除
blob_client.delete_blob(f"source/{doc_id}.pdf")
# 削除監査ログ記録（誰が・いつ・何を）
audit_log.record(action="delete", target=doc_id, requester=user_id)
```

**【なぜそうなのか】** ベクトル埋め込みは原文の意味情報を保持するため「匿名化済み」とは見なされない（EDPBガイドライン2024）。削除請求時は関連チャンク全削除＋再インデックス監査証跡まで求められる。設計段階で削除可能な構造にしないと、後追いでは技術的に困難。

> **📋 情シス向け説明フレーム**  
> 「ベクトルDBも個人データ保管庫として扱い、削除請求に応じられる設計にしています。削除キー設計・削除API・監査ログの3点セットが必須です。」

### Q2. Managed Identityとサービスプリンシパルの使い分けは？

**【背景】** 「APIキー運用は廃止し、全てMIに移行せよ」という社内方針が出る一方、外部SaaS連携やCI/CD・オンプレからのアクセス等、MIが使えないケースも残る。使い分け基準が不明瞭で現場が混乱。

**【推奨対策】**

| ケース | 推奨 | 理由 |
|---|---|---|
| Azure内リソース間通信（App Service → OpenAI） | **Managed Identity** | シークレット管理不要・自動ローテーション |
| オンプレ・他クラウドからAzureへ | サービスプリンシパル + 証明書認証 | MIは利用不可。証明書ベースでシークレット漏洩リスク軽減 |
| GitHub Actions等のCI/CD | **Workload Identity Federation** | シークレット保存不要。OIDC連携でトークン発行 |
| 短命スクリプト・開発者個人利用 | Azure CLI（`az login`） | 人間の認証を使う |

**【なぜそうなのか】** MIはAzureが自動でシークレットをローテーションするため、漏えい時のダメージを構造的に最小化できる。サービスプリンシパル＋クライアントシークレットは「最後の手段」。証明書認証またはWorkload Identity Federationで可能な限りシークレットレスにする。

> **📋 情シス向け説明フレーム**  
> 「シークレット管理事故の99%は人為ミス。構造的に保持させない設計（MI優先）が最も強固です。」

### Q3. RAGで部門横断検索と権限境界の両立は？

**【背景】** 「人事DBと法務DBを同一RAGで検索したい、ただし人事担当が法務機密を見られてはならない」という要件が頻出。ベクトル検索は本質的に全文横断のため、権限境界の実装設計が難しい。越境事故は1件でも重大コンプライアンス違反となる。

**【推奨対策】** **メタデータフィルタの強制（Security Trimming）** が唯一の堅牢な解。

```python
# 検索時、ユーザーのABAC属性を必ずフィルタに埋め込む
user_departments = get_user_departments(user_id)  # ["HR"]
user_clearance = get_user_clearance(user_id)  # "L3"

results = search_client.search(
    query_text=query,
    filter=f"department/any(d: search.in(d, '{','.join(user_departments)}')) "
           f"and clearance_level le {user_clearance}"
)
```

**設計上の3原則:**
1. インデックス作成時に全チャンクへ `department` / `clearance_level` / `allowed_roles` を必ず付与
2. 検索フィルタはアプリ側で強制注入（ユーザー指定不可）
3. フィルタ漏れを検知する定期監査クエリ（属性なしチャンクの検出）

**【なぜそうなのか】** LLMに「法務情報は人事に見せないで」と指示するだけでは防げない（Indirect PIで迂回可能）。検索フェーズで物理的に結果を絞り込む「取れない設計」が唯一の解。

> **📋 情シス向け説明フレーム**  
> 「権限境界はLLMの判断ではなく検索フィルタで物理的に保証します。LLMが見ていない情報は漏えいしようがありません。」

### Q4. PIIを学習/埋め込みに混入させない実装パターンは？

**【背景】** 匿名化を厳しくすると検索品質が落ちる、緩くすると漏えいリスクが残るというトレードオフに悩む企業が多い。「Azureは学習に使わないから大丈夫」という誤解も多いが、ログ・キャッシュ・ベクトルDBには残る。

**【推奨対策】** **3段階マスキング**でパイプライン全体を保護する。

| 段階 | 処理 | ツール例 |
|---|---|---|
| 取り込み時 | PII検出→トークン化（可逆）or マスキング（不可逆） | Microsoft Presidio / Azure AI Language PII |
| 検索時 | クエリ内のPIIも検出・マスキング | 同上 |
| 出力時 | LLM出力を再スキャン・PIIが含まれればマスキング | 同上＋正規表現 |

**トークン化 vs マスキング の選択基準:**
- 可逆トークン化（`[PERSON_001]`）: 後工程で復元が必要な業務（顧客対応等）
- 不可逆マスキング（`***`）: 復元不要な業務（分析・要約等）
- **要配慮個人情報（健康・信条・犯歴等）は原則LLM入力禁止**

**【なぜそうなのか】** 「Azureは学習に使わない」は正しいが、運用ログ・プロンプトキャッシュ・ベクトル埋め込みには残存する。PIIがそれらに混入した時点で情報漏えいリスクが発生する。入力前マスキングが最も確実。

> **📋 情シス向け説明フレーム**  
> 「PII保護は3段階（取り込み・検索・出力）で実装。『学習されない＝安全』ではなく『そもそも送らない』設計です。」

### Q5. ABAC属性の管理主体と棚卸頻度は？

**【背景】** ABAC（属性ベースアクセス制御）を導入すると柔軟性は上がるが、属性が乱立し「誰がこの属性を管理しているか不明」「属性が古くて権限が過剰/不足」という状態に陥りやすい。管理主体と棚卸プロセスの未整備が最大の失敗要因。

**【推奨対策】**

| 属性カテゴリ | 管理主体 | 棚卸頻度 | 連携先 |
|---|---|---|---|
| ユーザー属性（部門・役職） | 人事 | 月次（人事異動連動） | Entra ID / HRIS |
| データ属性（機密度・オーナー） | データオーナー（事業部） | 四半期 | Microsoft Purview |
| 環境属性（IPアドレス・時刻） | 情シス | 年次 | Conditional Access |
| プロジェクト属性 | PMO | プロジェクト開始/終了時 | Entra ID Groups |

**棚卸プロセスの要点:**
- 自動化: HRIS連動で退職者属性を即日無効化
- 可視化: 属性ごとに「最終使用日」を記録し、90日未使用属性は廃止候補
- 責任の明確化: 属性1つに必ず1名の管理者（RACI-Responsible）を任命

**【なぜそうなのか】** ABACの柔軟性は棚卸が前提。属性が管理されていないと、退職者や異動者の過剰権限が放置される。結果RBACより脆弱になるケースもある。

> **📋 情シス向け説明フレーム**  
> 「ABAC属性は『資産』として管理します。オーナーのいない属性は即廃止、最終使用日で自動棚卸するプロセスを組んでいます。」

## 第2部: 施策の戦略と実行

### フェーズ計画（RACI・Go基準付き）

| フェーズ | 期間 | 主な実施事項 | 実行(R) | 承認(A) | Go基準 |
|---|---|---|---|---|---|
| P1: データ分類・属性設計 | 2〜3週 | 機密度ラベル定義・ABAC属性カタログ作成 | データオーナー | CDO | 全データの分類完了 |
| P2: Managed Identity移行 | 2〜4週 | APIキー排除・MI/WIF切替 | 情シス・開発 | CISO | APIキー残数ゼロ |
| P3: RAG権限境界実装 | 3〜5週 | メタデータフィルタ強制・監査クエリ整備 | 開発チーム | 情シス | 越境テスト全件拒否 |
| P4: PIIマスキング実装 | 2〜3週 | Presidio導入・3段階マスキング設計 | 開発チーム | データ保護責任者 | PII検出率 > 95% |
| P5: 削除プロセス整備 | 2週 | 削除API・監査ログ・テナント削除手順 | 情シス | DPO | GDPR対応訓練完了 |
| P6: 定常運用 | 継続 | 月次属性棚卸・四半期権限監査 | 人事・情シス | CISO | 四半期監査レポート提出 |

### 成功の要諦（外せない3点）

1. **削除可能な設計を最初から組む** — 後追いでGDPR対応は技術的に困難。削除キー・削除API・監査ログを初期実装必須
2. **権限境界は検索フィルタで物理保証** — LLMの指示ベースでは必ず迂回される。メタデータフィルタ強制が唯一解
3. **属性を資産として管理** — ABAC属性は「作って終わり」ではない。オーナー・棚卸頻度・廃止基準を初日に決める

### よくある失敗パターン

- **ベクトルDB＝検索キャッシュ扱い:** 削除権対応が後手に回り、個人情報保護法違反リスクが累積
- **APIキーの「暫定」運用長期化:** 暫定のまま本番運用され、漏えい時に被害が拡大
- **属性乱立:** 類似属性が増え、管理不能化。結果権限過剰が常態化

## 第3部: よくある誤解と補足知識

### よくある誤解トップ5

| 誤解 | 正しい理解 |
|---|---|
| 「ベクトル埋め込みは匿名化済み」 | EDPBは個人データとみなす。削除対象 |
| 「Azureは学習に使わないから安全」 | ログ・キャッシュ・ベクトルDBに残存。入力前マスキングが必要 |
| 「LLMに『見せないで』と指示すれば防げる」 | Indirect PIで迂回可能。検索フィルタで物理保証が必須 |
| 「RBACで十分、ABACは複雑すぎる」 | 部門横断RAG等の要件ではABAC必須。ただし属性管理プロセスとセット |
| 「APIキーは環境変数なら安全」 | リポジトリ誤コミット事故が多発。MI/WIFへ移行が業界標準 |

### 例え話で理解する

> **RAGの権限境界は「図書館の立入禁止エリア」**  
> 人事資料と法務資料が同じ建物（ベクトルDB）にある場合、「見てはいけない」という貼り紙（LLMへの指示）では防げない。入館証（ABAC属性）で物理的に扉が開かない設計（メタデータフィルタ）が唯一の解。

### 用語集

| 用語 | 説明 |
|---|---|
| Managed Identity (MI) | Azureが自動でシークレット管理する認証方式 |
| Workload Identity Federation | OIDC連携でシークレットレスにトークン発行する方式 |
| ABAC | 属性ベースアクセス制御（部門・機密度等で動的判定） |
| Security Trimming | 検索フィルタで権限外データを物理的に除外する方式 |
| Microsoft Presidio | OSS のPII検出・匿名化ライブラリ |
