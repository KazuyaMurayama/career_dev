## 第1部: 実行フェーズの詳細化（重要3フェーズに集中）

Part 1 のフェーズ計画 P1〜P6 のうち、**最も躓きやすい3つ**を Day単位で詳細化。P2/P5/P6 は要点のみ。

### 深掘り①: P1 脅威モデリング（1〜2週）

「実装に入る前に何を守るかを決める」フェーズ。スキップすると後で必ず手戻り。

**Week1（実施タスクの分解）**

| Day | 実施事項 | 成果物 | 主担当 |
|---|---|---|---|
| Day1-2 | 業務フロー図とデータフロー識別 | システム構成図・データフロー図 | AIエンジニア |
| Day3 | OWASP LLM Top 10 × 自社システム ギャップ分析 | ギャップマトリクス | セキュリティ担当 |
| Day4 | STRIDE-LM 拡張で脅威列挙（Spoofing〜Lateral Movement） | 脅威リスト（20〜40件） | セキュリティ担当 |
| Day5 | リスクスコアリング（影響度×発生可能性） | リスク登録簿 | セキュリティ担当 + PL |

**Week2（合意形成）**
- リスク一覧レビュー会（事業部門・情シス・CISO）
- 経営承認取得（高リスク項目の対応方針合意）

**STRIDE-LM 拡張の具体例**

| 脅威カテゴリ | LLM固有の脅威例 | 主防御 |
|---|---|---|
| Spoofing | プロンプトなりすまし（"管理者として..."） | システムロール固定・XMLタグ分離 |
| Tampering | RAG文書改ざん（Indirect PI） | 取得源検証・整合性ハッシュ |
| Repudiation | "AIが言った"の否認 | request_id監査ログ（Part 3 Q1） |
| Information Disclosure | PII・機密漏えい | Presidio・Security Trimming（Part 2） |
| Denial of Service | トークン消費攻撃 | レート制限・トークン上限 |
| Elevation of Privilege | Jailbreakによる権限昇格 | Content Filter + 自前ガードレール |
| **Lateral Movement (LLM拡張)** | エージェント連鎖の暴走 | 各ツール呼出に承認ゲート |

**典型的な躓き**
- 脅威モデリングをスキップして実装に入る → ガードレールの過不足が判明し、本番直前に大幅手戻り
- リスクスコアの主観性 → 影響度・発生可能性とも 1〜5 の数値定義表を事前合意

### 深掘り②: P3 レッドチーミング（1〜2週）

「実装したガードレールが本当に効くか検証する」フェーズ。自動 + 人間の両輪が必須。

**Week1（攻撃テスト実施）**

| Day | 実施事項 | ツール・手順 |
|---|---|---|
| Day1 | 攻撃シナリオ定義（10〜20本） | OWASP LLM01〜10から重要5カテゴリ + 業務固有3〜5本 |
| Day2-3 | Garak 自動実行 | CI/CDパイプラインに組込み、`probes: dan, jailbreak, promptinject` を優先 |
| Day4-5 | 人間レッドチーム（半日×3名） | 自動では出ない多段攻撃・社内文脈攻撃を試行 |

**Garak の CI/CD 組込み例**

```yaml
# .github/workflows/redteam.yml
- name: Run Garak red team scan
  run: |
    garak --model_type openai --model_name gpt-4o \
          --probes dan,jailbreak,promptinject,encoding \
          --report_prefix redteam_$(date +%Y%m%d)
- name: Fail if ASR exceeds threshold
  run: python check_asr.py --threshold 0.05
```

**業務固有シナリオの作り方（3〜5本）**
- 「自社の最重要文書」を Indirect PI で漏えいさせる試行
- 「顧客対応で絶対避けたい発言」を Jailbreak で引き出す試行
- 「システムプロンプト全文」を間接的に取得する試行（ロールプレイ・翻訳・1文字ずつ等）

**Week2（修正と再テスト）**
- ASR > 5% の項目について修正チケット起票（Part 1 Q5 準拠）
- 修正後の再テスト → ASR 改善確認
- 教訓集を Wiki に記録（次四半期の演習用）

**典型的な躓き**
- 自動ツールのみで人間レッドチーム省略 → 業務固有攻撃が漏れる
- 1回テストして満足 → 攻撃手法は日々進化、四半期定例化が必須（Part 1 Q5）

### 深掘り③: P4 ステージング閾値調整（2週）

「本番影響なく実データで閾値を最適化する」フェーズ。FP率を下げないと「使えないAI」評価で予算削減。

**Week1: シャドー運用**
- 本番LLM呼び出しと並行で、ガードレール判定のみ記録（出力には影響させない）
- 並行監視4指標: ASR / FP率 / P95レイテンシ / ユーザー満足度
- 1週間で 1,000〜10,000 件の実トラフィックを収集

**Week2: 閾値の段階調整**

| 項目 | 調整手順 |
|---|---|
| Content Filter severity | low/medium/high の3段階を AB テスト、FP率と検知率のバランス点を見つける |
| 自前ガードレール正規表現 | FPケースを抽出 → 表現を絞り込み → 再テスト |
| Groundedness 閾値 | 0.5/0.7/0.9 の3水準を比較、業務インパクトと検知率のトレードオフ確認 |

**FPケース集の作り方**
- 誤検知された入力を匿名化し Wiki にカテゴリ別蓄積
- 月次で「新規FPパターン」を抽出 → 閾値再調整ループ

**典型的な躓き**
- いきなり本番で閾値調整 → 顧客影響発生
- FP率を放置 → 「正常な質問もブロックされる」と評価され利用率低下

### その他フェーズ（要点のみ）

| フェーズ | 最重要ポイント |
|---|---|
| P2 PoC実装 | XMLタグ分離は初期実装必須。後付け困難 |
| P5 本番リリース | 段階ロールアウト（1% → 10% → 100%）+ Azure Monitor アラート設定 |
| P6 定常運用 | 四半期レッドチーム + モデル更新時の再評価をカレンダー化 |

## 第2部: 業界別適応例（5業界×最重要打ち手）

各業界の**最も致命的なリスク1つ**に絞り、Part 1 の打ち手をどう適応するかを示す。

### 製造業

| 項目 | 内容 |
|---|---|
| 最重要リスク | 設計図・製造ノウハウ（営業秘密）の RAG 混入 → 競合への漏えい |
| 関連法規 | 不正競争防止法・営業秘密管理指針（経産省） |
| 最優先シナリオ | 海外拠点エンジニアが社内RAGに質問 → 国内のみ閲覧許可の図面を取得 |
| 核心打ち手 | 図面メタデータに「技術機密度」「閲覧可能拠点」のABAC属性付与 + Security Trimming（Part 2 Q3） |
| Part 1 補強 | RAG取得元の地理属性チェックをガードレールに追加（拠点コードがフィルタを通過しなければ強制ブロック） |

### 金融業

| 項目 | 内容 |
|---|---|
| 最重要リスク | 個人信用情報のLLM送信・ハルシネーションによる商品誤案内 |
| 関連法規 | 銀行法・金商法・FISC 安全対策基準・個人情報保護法 |
| 最優先シナリオ | 顧客対応AIが投資商品の利回りを誤回答 → 顧客損失・適合性原則違反 |
| 核心打ち手 | 出力前 Groundedness チェック必須化（閾値 > 0.85）+ FISC 準拠 WORMログ（Part 3 Q1） |
| Part 1 補強 | 金融商品・利回り・契約条件の言及は **必ず引用元を表示**、引用なしの数値は強制ブロック |

### 製薬業界

| 項目 | 内容 |
|---|---|
| 最重要リスク | GxP対象データのAI処理・治験情報漏えい・規制当局報告の誤り |
| 関連法規 | GxP（GLP/GCP/GMP）・21 CFR Part 11（FDA）・薬機法 |
| 最優先シナリオ | 治験データ要約AIが患者識別情報を出力 → 21 CFR Part 11 違反 |
| 核心打ち手 | プロンプト・出力の**完全監査証跡**（電子署名付き）+ PII3段階マスキング（Part 2 Q4） |
| Part 1 補強 | GxP対象データ処理は専用デプロイメントを分離、汎用LLMから物理的に隔離 |

### EC（小売・通販）

| 項目 | 内容 |
|---|---|
| 最重要リスク | 顧客チャットの誤発言（Air Canada判例リスク） |
| 関連法規 | 消費者契約法・特定商取引法・景品表示法 |
| 最優先シナリオ | チャットbot が割引・返品条件を誤案内 → 履行義務発生・SNS炎上 |
| 核心打ち手 | 価格・返品・契約条件の回答は **RAG + Human-in-the-loop** + 出力前ポリシーチェック |
| Part 1 補強 | 「割引」「返金」「キャンセル」等のキーワード検知時はゲートで人間承認必須 |

### インフラ（電力・通信）

| 項目 | 内容 |
|---|---|
| 最重要リスク | OT/ICS 情報のAI混入・重要インフラ規制違反 |
| 関連法規 | 経済安全保障推進法（基幹インフラ）・電気事業法・電気通信事業法 |
| 最優先シナリオ | 社内RAGに保守AIがアクセス → 制御系の脆弱性情報を出力 → サイバー攻撃に悪用 |
| 核心打ち手 | OT/ICS データは Private Endpoint で IT系から完全分離 + クリティカル出力の二重承認ゲート |
| Part 1 補強 | 「制御」「PLC」「SCADA」等のキーワード検知時は出力を強制マスキング、CSIRT通知 |

## 第3部: 優先順位付けサマリー

### 全業界共通の最優先3打ち手

複数業界の検討で共通して上位に来る打ち手。これだけは**全AI導入企業が必ず実施**すべき。

1. **Indirect PI 対策の XMLタグ分離**（Part 1 Q1）
   - RAG/エージェント運用の事実上前提
   - 構造的分離で言い換え攻撃も防御可能
2. **出力前サニタイズの二重化**（Part 1 Q2）
   - LLM基盤フィルタ + アプリ層 DOMPurify
   - XSS・SQLインジェクション・SSRFを多層で防御
3. **業務固有ガードレール + ベンダーフィルタの2層構成**（Part 1-1 第2部）
   - ベンダーロックイン回避と業務リスクカバーを両立
   - request_id で監査ログを統合

### 業界別「上乗せ1打ち手」クイック参照

| 業界 | 上乗せ打ち手 |
|---|---|
| 製造業 | ABAC属性「技術機密度・閲覧拠点」の Security Trimming |
| 金融業 | Groundedness 閾値 > 0.85 + FISC準拠 WORMログ |
| 製薬 | プロンプト・出力の電子署名付き完全監査証跡 |
| EC | 価格・契約条件回答の Human-in-the-loop 必須 |
| インフラ | OT/ICS データの Private Endpoint 物理分離 + 二重承認 |

### 実行順序の推奨

1. **共通3打ち手を全社で先行実装**（フェーズF1: 〜2025Q4、Part 5 Q3参照）
2. **業界別上乗せを各事業部門で展開**（フェーズF2〜F3）
3. **四半期レッドチーム + 業界規制更新監視を定例化**（フェーズF4〜）

> **📋 情シス向け説明フレーム**  
> 「共通3打ち手で全社の最低ラインを敷き、業界別1打ち手で事業特性に適応します。優先順位を絞ることで、半年以内に最大リスクの80%をカバー可能です。」

## 補足: よくある実行上の誤解

| 誤解 | 正しい理解 |
|---|---|
| 「全フェーズを均等に詳細化すべき」 | 重要3フェーズ（脅威モデリング・レッドチーム・閾値調整）に紙面集中が現実解 |
| 「業界共通ベストプラクティスで十分」 | 最重要リスクは業界で異なる。共通+業界別1打ち手の二段構えが必須 |
| 「FP率は本番運用後に調整」 | ステージング段階で実データシャドー運用が必須。本番調整は顧客影響発生 |
