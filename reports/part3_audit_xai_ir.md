## 第1部: 重要Q&Aベスト5

### Q1. LLMログに何を残すべきか、プロンプト全文保存は是か非か？

**【背景】** LLM運用ログ設計は従来のWebアプリ監査ログと要件が異なる。プロンプト全文保存はフォレンジック価値が高い反面、機密プロンプトの長期保管自体がリスク。「全部残す」でも「最小限」でもなく、機密度別の保管方針が必要。

**【推奨対策】** 監査要件とリスクを両立する**ハイブリッドスキーマ**を採用。

| ログ項目 | 保存 | 保管期間 | 備考 |
|---|---|---|---|
| request_id / user_id / timestamp | 必須 | 7年（金融）/ 3年（一般） | 追跡の起点 |
| プロンプトのハッシュ値 | 必須 | 同上 | 改ざん検知 |
| プロンプト全文 | **機密度別** | 30〜90日 | Confidential以上はマスキング後保存 |
| 取得RAG文書ID | 必須 | 同上 | ソース追跡 |
| LLM出力全文 | 必須 | 同上 | インシデント調査用 |
| トークン数・モデルバージョン | 必須 | 同上 | コスト・ドリフト分析 |
| Content Filter判定結果 | 必須 | 同上 | 検知統計 |

**WORM（Write Once Read Many）保存必須:**
- Azure Immutable Blob Storage / AWS S3 Object Lock
- 改ざん不可・削除不可を設定で強制

**【なぜそうなのか】** プロンプト全文を永年保存すると、漏えい時の被害が倍増する。一方、全く保存しないとインシデント調査が不能。「機密度別の短期保存 + ハッシュの長期保存」で両立できる。WORM は EU AI Act・FISC の改ざん防止要求にも対応。

> **📋 情シス向け説明フレーム**  
> 「プロンプトは機密度別に保存期間を分け、全件ハッシュで改ざん検知、WORM保存で改ざん不可を構造的に保証しています。」

### Q2. RAG回答のソース提示不十分と監査指摘された場合の是正策は？

**【背景】** EU AI Act（透明性要求）や金融庁・医療ガイドラインでは「AI判断の根拠提示」が義務。RAGで回答しても「どの文書の何ページから引用したか」が曖昧だと監査指摘を受ける。単なる文書名表示では不十分。

**【推奨対策】** **引用3点セット**を全回答に必須化。

```json
{
  "answer": "就業規則では有給休暇は...",
  "citations": [
    {
      "source": "就業規則_v2.3.pdf",
      "page": 12,
      "chunk_id": "chunk_1234",
      "similarity_score": 0.89,
      "excerpt": "第18条 有給休暇は..."
    }
  ],
  "confidence": 0.85,
  "retrieved_at": "2026-04-22T10:30:00Z"
}
```

**UI実装の要点:**
- 回答の各段落に引用番号 [1] [2] を付与
- クリックで原文該当箇所をハイライト表示
- 類似度スコアが閾値未満（例 < 0.7）の場合は「根拠が弱い」警告表示

**【なぜそうなのか】** EU AI Act第13条は「ユーザーが出力を理解・検証できる情報提供」を義務付ける。文書名だけでは検証不能。ページ・チャンクID・抜粋まで提示することで、ユーザーが一次ソースに辿り着ける設計が必要。

> **📋 情シス向け説明フレーム**  
> 「全回答に引用3点セット（文書・ページ・抜粋）と信頼度スコアを必須表示。ユーザーが一次ソースで検証できる設計です。」

### Q3. 幻覚（hallucination）による誤情報はインシデントか否か？

**【背景】** ハルシネーションは日常的に発生するため「毎回インシデント扱いは現実的でない」一方、顧客に誤情報が伝わった場合は明確にインシデント。判定基準が未整備だとエスカレーション判断が属人化する。

**【推奨対策】** **3段階トリアージ**で判定を自動化する。

| 重大度 | 基準 | 対応 |
|---|---|---|
| Critical | 顧客に誤情報が実害（金銭・健康・法的）として到達 | 即時インシデント・CSIRT起動・PR検討 |
| High | 社外公開済み / 誤情報が広範囲流通 | インシデント起票・24h以内報告 |
| Medium | 社内のみ・個別ユーザーに誤情報 | 該当回答訂正・プロンプト改善 |
| Low | 検知済みで自動訂正 | 月次レポートで集計 |

**判定フロー:**
1. 出力にRAG根拠を必須付与（Q2参照）→ 根拠との整合性スコア算出
2. 整合性スコア < 0.5 の回答は自動フラグ立て
3. 社外公開チャネル（メール送信・PDF生成）の前に必ず人間レビューゲート

**【なぜそうなのか】** ハルシネーションは0にはできない確率現象。「発生させない」ではなく「発生しても影響を局所化する」設計が実務解。社外到達前の人間ゲート設置が最大の防御。

> **📋 情シス向け説明フレーム**  
> 「ハルシネーションは0にできないため、社外到達前の人間ゲートで被害を局所化する設計。重大度Critical以上は即CSIRT起動します。」

### Q4. モデルドリフト検知のSLOと閾値設定は？

**【背景】** 本番運用のLLM/MLモデルは、入力分布の変化や API側のモデル更新で精度が劣化する。放置すると誤回答が増え、気付いた時には顧客影響が拡大。SLO（Service Level Objective）と閾値設定が未整備の企業が多い。

**【推奨対策】** **4指標のSLO**を定義し、ダッシュボードで常時監視。

| 指標 | 定義 | 警告閾値 | 是正閾値 |
|---|---|---|---|
| ハルシネーション率 | RAG整合性 < 0.5 の割合 | > 3% | > 5% |
| ユーザー満足度 | 👍/👎フィードバックの正率 | < 85% | < 80% |
| 平均応答時間 | P95レイテンシ | > 5秒 | > 10秒 |
| 入力分布KL距離 | 学習時分布との乖離 | > 0.3 | > 0.5 |

**アラート設計:**
- 警告閾値超過 → Slack通知・担当者レビュー
- 是正閾値超過 → CSIRT起動・プロンプト/モデルロールバック検討
- 月次レビュー → トレンド分析・閾値再調整

**【なぜそうなのか】** ドリフトは段階的に進行するため、単一閾値では手遅れになりやすい。「警告→是正」の2段階閾値で早期検知と自動対応を両立できる。NIST AI RMF MEASURE機能の中核要件。

> **📋 情シス向け説明フレーム**  
> 「4指標のSLOをリアルタイム監視。警告閾値で予防対応、是正閾値で自動エスカレーション。月次レビューで閾値を継続改善します。」

### Q5. AIインシデントのCSIRTエスカレーション経路と証拠保全は？

**【背景】** 従来のCSIRT手順はマルウェア・不正アクセスが前提で、AI固有インシデント（プロンプトインジェクション・モデル異常挙動・エージェント超過行動）に対応できない。エスカレーション経路と証拠保全手順が未整備だと初動が遅れる。

**【推奨対策】** **AI固有のエスカレーション経路**を従来CSIRTに追加。

```
[検知]
 監査ログアラート / ユーザー報告 / 定期スキャン
    ↓
[トリアージ（30分以内）]
 AIセキュリティ担当 → 重大度判定（Critical/High/Medium/Low）
    ↓
[証拠保全（即時）]
 - request_id 連結で全プロンプト・出力・取得文書をスナップショット
 - WORMストレージへコピー（改ざん防止）
 - モデルバージョン・Content Filter設定を記録
    ↓
[封じ込め]
 Critical/High → 即時サービス停止 or デプロイメント切替
 Medium/Low   → 該当ユーザー封鎖 + 監視強化
    ↓
[調査・根本原因分析]
 OWASP LLM Top10 のどのカテゴリか判定
    ↓
[報告]
 社内AIガバナンス委員会（Medium↑: 24h以内）
 CISO/CDO（Critical: 即時 / High: 2h以内）
 個人情報漏えい時: 72h以内に個人情報保護委員会（法定）
```

**証拠保全の必須項目（ISO/IEC 27035準拠）:**
- プロンプト全文・出力全文・RAG取得文書ID・モデルバージョン・Content Filter結果・ユーザー属性

**【なぜそうなのか】** AI固有インシデントは「request_id」を軸にした横断追跡が唯一の調査手段。従来の「サーバーログ」発想では復元不能。WORM保存と同時に、AI特有のメタデータを保全する設計が必須。

> **📋 情シス向け説明フレーム**  
> 「AI固有IR手順を従来CSIRTに追加し、request_id軸でのスナップショット保全を即時実行。法定報告タイミングも手順に組み込み済みです。」

## 第2部: 施策の戦略と実行

### フェーズ計画（RACI・Go基準付き）

| フェーズ | 期間 | 主な実施事項 | 実行(R) | 承認(A) | Go基準 |
|---|---|---|---|---|---|
| P1: ログスキーマ設計 | 2週 | 機密度別保存方針・WORM設計 | AIエンジニア | 情シス・CISO | スキーマ承認 |
| P2: 引用・XAI実装 | 3〜4週 | 引用3点セット・信頼度表示 | 開発チーム | 事業部門 | 全回答に引用付与 |
| P3: ドリフト監視 | 2〜3週 | SLO定義・ダッシュボード構築 | AIエンジニア | PL | 4指標可視化完了 |
| P4: IR手順整備 | 3週 | AI固有IR手順書・証拠保全手順 | CSIRT+AI担当 | CISO | 机上演習合格 |
| P5: 机上演習 | 1週 | 3シナリオ演習（PI・漏えい・超過行動） | 全関係者 | CISO | 全シナリオ対応完了 |
| P6: 定常運用 | 継続 | 月次SLOレビュー・四半期IR演習 | CSIRT | CISO | 月次レポート提出 |

### 成功の要諦（外せない3点）

1. **request_id を全レイヤーに貫通させる** — 入力・RAG取得・出力・フィルタ結果を同一IDで追跡可能にする。これが無いと事後調査は不能
2. **引用とXAIは後付け不可** — 初期実装に組み込まないと、本番稼働後の後追いは技術的に困難
3. **机上演習を定例化** — 手順書だけでは動けない。四半期1回のシナリオ演習で初動速度を維持

### よくある失敗パターン

- **ログ設計の後回し:** 本番稼働後の追加は困難。初日からWORM設計必須
- **引用の文書名のみ表示:** EU AI Act・金融庁要件を満たさない。ページ・抜粋まで必須
- **ハルシネーション基準未整備:** 重大度判定が属人化し、重大事案の見逃しが発生

## 第3部: よくある誤解と補足知識

### よくある誤解トップ5

| 誤解 | 正しい理解 |
|---|---|
| 「プロンプト全文を永年保存すべき」 | 機密漏えいリスクが倍増。ハッシュ長期+全文短期のハイブリッドが解 |
| 「ハルシネーションは全部インシデント」 | 3段階トリアージで社外影響のみCritical扱いが実務解 |
| 「ドリフトは単一閾値で十分」 | 警告→是正の2段階閾値で早期検知と自動対応を両立 |
| 「引用は文書名表示で十分」 | EU AI Act/金融庁はページ・抜粋・スコアまで要求 |
| 「従来CSIRT手順でAIもカバーできる」 | AI固有手順（request_id追跡・WORM保全）の追加が必須 |

### 例え話で理解する

> **AI監査ログは「航空機のフライトレコーダー」**  
> 事故が起きた時、コックピット音声（プロンプト）・計器データ（モデルバージョン・トークン数）・管制交信（RAG取得）が同期していないと原因究明は不能。request_id が時刻同期信号、WORMが耐熱耐衝撃の筐体。

### 用語集

| 用語 | 説明 |
|---|---|
| WORM | Write Once Read Many。改ざん・削除不可の保存方式 |
| ドリフト | 入力分布やモデル挙動が時間経過で変化すること |
| SLO | Service Level Objective。運用品質の定量目標 |
| KL距離 | 確率分布の乖離を測る指標（ドリフト検知で使用） |
| CSIRT | Computer Security Incident Response Team |
