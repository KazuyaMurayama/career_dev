# Timeout Prevention — Claude Code 実行ガイド

## 根本原因

Claude の1レスポンスあたりの出力トークン上限（約 8,192 tokens）が制約。  
`push_files` や `create_or_update_file` はファイル全文をパラメータ文字列として出力するため、ファイルが大きいほどトークンを消費する。

**このセッションで発見した閾値（経験則）：**

| ファイル行数 | トークン概算 | 結果 |
|---|---|---|
| ~200行以下 | ~5,000 tokens | ✅ 成功 |
| ~275行 | ~6,500 tokens | ✅ 成功（要注意） |
| ~365行以上 | ~8,000 tokens超 | ❌ タイムアウト |
| ~625行（10章結合） | ~13,000 tokens | ❌ 根本的に不可能 |

---

## 有効だった手法（優先順）

### 手法1: `create_or_update_file` を使う（`push_files` より低オーバーヘッド）

```
# 推奨
create_or_update_file → 1ファイル専用、呼び出しコスト小
push_files            → 複数ファイル一括だがオーバーヘッド大
```

275行の結合ファイルを `create_or_update_file` で成功、同内容を `push_files` では失敗した実績あり。

### 手法2: 200行を目安に1ファイル = 1回 push で完結させる

- 1ページ ≒ 15〜20行（ヘッダー・表・コードブロック込み）
- 14ページのレポート ≒ 200〜280行 → **200行以内に絞る設計を先行**
- 超える場合は必ず手法3へ

### 手法3: 2分割一時ファイル → 結合パターン（200〜280行のファイル向け）

```
タスク1: _tmp/report_a.md（前半 ~110行）を push
タスク2: _tmp/report_b.md（後半 ~90行）を push
タスク3: 両方を読み込み → report.md として結合 push（200行）
タスク4: _tmp/ 以下を削除
```

⚠️ 結合 push 自体が200行を超えないよう設計すること。

### 手法4: 並列読み込み → 逐次書き込み

- `get_file_contents` は複数ファイルを1メッセージで並列実行 → 入力トークンに影響なし
- `create_or_update_file` / `delete_file` は **逐次実行**（並列実行するとブランチHEADの競合でどちらかが失敗する）

```
# 良い例
読み込み: [file_a, file_b, file_c] を同時に get_file_contents

# 書き込み
push file_a → 完了を確認 → push file_b → 完了を確認 → push file_c
```

### 手法5: ファイルを再生成しない（読み取り内容をそのまま使う）

- 一度 `get_file_contents` で取得した内容はコンテキストに保持される
- 結合時は「再生成」せず「読み取った文字列をそのまま連結」してパラメータに渡す
- 再生成するとトークンが2倍消費されタイムアウトリスクが上がる

---

## 失敗パターン（やってはいけないこと）

| パターン | 理由 |
|---|---|
| 10章を1ファイルに結合しようとする | 625行 ≒ 13,000 tokens、どうやっても不可能 |
| 段階的に append しようとする | GitHub API に append 操作は存在しない。毎回全文が必要 |
| 書き込みを並列実行する | ブランチHEADの競合でどちらかが `422 Conflict` エラー |
| 大きなファイルを `push_files` で送る | `create_or_update_file` より overhead が大きい |
| 内容を再生成しながら結合する | トークン消費2倍、不必要なリスク |

---

## 実行前チェックリスト

```
□ 作成・更新するファイルは200行以内か？
□ 200〜280行の場合 → create_or_update_file を使うか？
□ 280行超の場合 → 2分割一時ファイルパターンに変更したか？
□ 複数ファイルの書き込みを並列実行していないか？
□ progress.md を確認して作業順を把握したか？
```
