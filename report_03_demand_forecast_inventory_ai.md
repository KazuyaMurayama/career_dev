# レポート3：需要予測・在庫管理のAI適用 ― 製造業特有の複雑性への対処

---

## 1. 製造業SCMの在庫3層構造

製造業の在庫はEC・小売と根本的に異なる。「売れた分だけ補充」の発想は通じない。

| 在庫層 | 内容 | 管理の難しさ |
|---|---|---|
| **原材料在庫** | 製造に使う材料・包材 | サプライヤーリードタイム・価格変動・最低発注量 |
| **仕掛品在庫** | 製造途中の中間品 | 工程の進捗・品質検査の待ち時間 |
| **製品在庫** | 出荷待ちの完成品 | 需要変動・賞味期限・保管コスト |

> **例え話：** EC（Amazon）の在庫管理は「冷蔵庫の食材を切らさない」こと。製造業は「農作物の種まき→収穫→加工→出荷」の全工程を管理するようなもの。前工程の遅れが後工程に連鎖し、需要予測のズレが原材料の過剰発注に直結する。

**太陽油脂の特性：**
- 油脂原料（パーム油・大豆油等）は国際商品市場で価格が変動
- 製造リードタイム：原料調達から製品完成まで3〜6週間
- 包材（容器・ラベル）は原料より先に発注が必要

## 2. 需要予測精度とリードタイムの関係

**なぜリードタイムが長いほど予測が難しいか：**
需要予測の誤差は時間軸が長くなるほど拡大する（不確実性の錐）。リードタイム4週の場合、誤差±20%でも在庫に大きな影響が出る。

| リードタイム | 予測精度要件 | 推奨バッファ設計 |
|---|---|---|
| 1〜3日（EC型） | 精度への感度が低い | 安全在庫＝σ×1.65（標準的） |
| 1〜2週（短リード製造） | ±15%以内が望ましい | 安全在庫＋緊急発注ルール |
| 3〜6週（太陽油脂型） | ±10%超でも大きな影響 | 安全在庫の厚め設定＋複数シナリオ予測 |

**バッファ設計の実務：**
- 原料リードタイム（仕入先→工場）
- 製造リードタイム（仕込み→完成）
- 出荷リードタイム（完成→顧客納品）
これら3つの合計がSCM全体のリードタイム。AIはこの構造を把握した上で設計する。

## 3. AIによる需要予測：適用可能領域と限界

### 適用可能（AIが得意）

| 領域 | 内容 | 推奨アプローチ |
|---|---|---|
| 定番品の需要予測 | 過去2〜3年の販売データがある商品 | 時系列予測モデル（Prophet/SARIMA） |
| 季節変動の自動補正 | 夏場需要増・年末需要増 | 機械学習モデルに季節変数を追加 |
| 販促イベント補正 | キャンペーン前後の需要変化 | 過去イベントデータを特徴量として投入 |

### 限界（人間の判断が必要）

| 限界領域 | 理由 | 対処 |
|---|---|---|
| 新規商品の需要設定 | 過去データが存在しない | 類似品の初期需要を参考にしつつ営業判断 |
| 原料価格急騰時の発注抑制 | 外部環境変化をAIが認識できない | アラートを出してプランナーが調整 |
| 競合品の突然の参入・撤退 | 市場構造変化 | 外部情報を組み込む仕組みが必要 |

## 4. MRP（製造資源計画）の仕組みとAI/RPAの切り口

**MRPの計算ロジック（基本）：**
```
必要量 = 製品需要予測 × 原料構成比（BOM）- 現在庫 - 発注残
```
※ BOM（Bill of Materials）＝製品1単位に必要な原料リスト

**MRP計算の自動化適性：**

| MRP業務 | 自動化適性 | AI/RPA適用内容 |
|---|---|---|
| ERPからの需要データ取得 | 即時自動化 | RPA（ERP操作自動化） |
| BOM×需要の計算 | 即時自動化 | 計算エンジン自動実行 |
| 在庫・発注残の照合 | 即時自動化 | ERPデータ自動参照 |
| 発注量の最終調整 | ハイブリッド | AI初期値 → プランナー承認 |
| 新規原料の調達判断 | 人間担当 | 担当者主導 |

## 5. 太陽油脂型：原料・包材MRP自動化の設計案

**現状（ASIS）の問題点：**
- 週1回の手動処理に2名×4時間 = 週8時間を消費
- ERPからExcelへの手動コピペで転記ミスが発生
- 担当者不在時は計算が止まる（属人化リスク）

**自動化設計（段階的）：**

フェーズ1（PoC・1ヶ月）：
- ERPからのデータ自動取得RPA開発
- Excelベースの計算エンジン自動化
- 結果レポートの自動生成

フェーズ2（本番・2ヶ月）：
- 夜間バッチ自動実行
- 異常値アラート自動通知（担当者メール）
- ERP自動登録（上長承認後）

フェーズ3（発展・6ヶ月〜）：
- AIによる需要予測値をMRP計算に自動反映
- 原料価格変動アラートとの連動

**期待効果：**
- 週8時間 → 週2時間（確認・承認のみ）
- ミス率：大幅低下（コピペ作業ゼロ化）
- 担当者を戦略業務（サプライヤー交渉・需要分析）に再配置

## 6. KPI設定（測定なくして改善なし）

| KPI | 定義 | 目標値（目安） |
|---|---|---|
| 需要予測精度（MAPE） | 予測と実績の平均絶対誤差率 | ±15%以内（製造業平均） |
| 在庫回転率 | 年間売上高 ÷ 平均在庫額 | 業種平均以上 |
| 欠品率 | 欠品件数 ÷ 総受注件数 | 1%以下 |
| MRP計算工数 | 月間MRP作業にかかる人時間 | ▲70%以上削減 |
| 安全在庫水準 | 平均在庫 ÷ 安全在庫比率 | 過剰在庫比率10%以内 |

## 失敗パターンと回避策

| 失敗パターン | 発生場面 | 回避策 |
|---|---|---|
| 需要予測AIを入れたが精度が出ない | データ品質が低い状態で導入 | データクレンジング（最低12ヶ月）を先行 |
| MRP自動化後に在庫過剰が発生 | バッファ設定を変えずに自動化 | 自動化と同時に安全在庫水準を見直す |
| 原料価格変動を予測に組み込めない | 外部データ連携を後回しにした | 初期設計で外部データ取得口を設けておく |
| 担当者がAI予測値を無視する | AI精度への不信感 | 初期3ヶ月はAI予測vs人間予測を並走させ精度を可視化する |

## 想定質問と回答

**Q1：「需要予測AIを入れたが精度が上がらない。原因は何か？」**

**質問の背景：** 導入後に期待値を下回るケースは多い。顧客（業務改革部・IT）が「なぜ効果が出ないか」を問い詰める局面。

**回答（推奨）：** 精度が上がらない原因は①データ品質（欠損・外れ値・データ期間不足）、②モデル選択の不一致（商品特性と予測手法がミスマッチ）、③外部変数の未組み込み（キャンペーン・市場変化）の3つが9割を占める。まずデータ診断を実施し、問題の所在を特定する。「モデルを変える」前に「データを直す」が先決。

他の選択肢との比較：モデルを高度化（ディープラーニング化）する方向は、データが整備されていない状態ではさらに精度が下がるリスクがある。

---

**Q2：「MRPとAI需要予測をどう組み合わせるか？どちらが主か？」**

**質問の背景：** 両者の役割分担が不明確なままシステム設計を進めると「2つの予測値がある」混乱が起きる。

**回答（推奨）：** 「AI需要予測が主、MRPは実行計算エンジン」が正しい役割分担。AIが将来の需要量を予測し、その数値をMRPに入力して必要な原材料量を計算する。MRPは「計算機」であり「予測機」ではない。両者の役割を明確にした上でシステム連携を設計する。

他の選択肢との比較：「MRP内の標準予測」を使い続けると、過去平均ベースの予測になり、季節変動・トレンド変化に対応できない。AI予測への切り替えが長期的にはROIが高い。

## 用語集

| 用語 | 説明 |
|---|---|
| MRP（製造資源計画） | 製品の生産計画から逆算して原材料の必要量・発注タイミングを計算する仕組み |
| BOM（Bill of Materials） | 製品1単位を作るために必要な原材料・部品のリストと量 |
| 安全在庫 | 需要変動・供給遅延に備えて保持する余剰在庫 |
| MAPE | Mean Absolute Percentage Error。需要予測精度の代表的指標 |
| 時系列予測モデル | 過去の数値の時間的パターンから将来を予測するAIモデル |
| 在庫回転率 | 在庫がどれだけ速く売れているかを示す効率指標 |
| 不確実性の錐 | 予測期間が長くなるほど誤差が拡大するという概念 |
| バッファ在庫 | リードタイムのばらつきに備えて積み増す在庫量 |
