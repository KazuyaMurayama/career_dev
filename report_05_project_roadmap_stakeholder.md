## 1. SCM変革プロジェクトの典型的フェーズ

| フェーズ | 期間（目安） | 主なアウトプット | 意思決定者 |
|---|---|---|---|
| **フェーズ0：診断** | 1〜2ヶ月 | ASIS調査・課題整理・初期仮説 | プロジェクトオーナー |
| **フェーズ1：設計** | 2〜3ヶ月 | TOBE設計・GAP分析・業務再設計案 | 業務改革部・経営幹部 |
| **フェーズ2：PoC** | 1〜2ヶ月 | 技術実証・効果測定・本番設計 | 業務改革部 |
| **フェーズ3：本番展開** | 3〜6ヶ月 | システム本番稼働・業務移行 | 役員会・全社 |
| **フェーズ4：定着** | 6ヶ月〜 | KPIモニタリング・改善 | 現場リーダー |

> **例え話：** BPRプロジェクトは「新工場を建てる」ことに似ている。設計図を描く前に土地（現状）を調べ（診断）、設計図（TOBE）を作り、模型で確認して（PoC）、実際に建てる（展開）。土地調査を省いた家は傾く。

## 3. 役員会プレゼン（9月20日）の「4点セット」

役員会で承認を取るために必要な最低限の要素：

| 要素 | 内容 | 太陽油脂の場合 |
|---|---|---|
| **①対象業務と削減効果** | 何をどう変えて何人削減できるか | 受注入力7→4名、配車手配・MRP等で追加4名削減 |
| **②投資額と回収期間（ROI）** | いくらかけていつ回収できるか | システム投資額 vs 人件費削減額の試算 |
| **③リスクと対策** | 何がうまくいかない可能性があるか | 技術リスク・現場抵抗・親会社合意リスクと対策 |
| **④段階承認フロー** | いつ何を決定するか | PoC完了後に本番展開を再承認する設計 |

**役員会プレゼンの構成（推奨15分）：**
1. エグゼクティブサマリー（3分）：結論→数字→お願い
2. 現状と課題（3分）：ASIS+問題の定量化
3. 変革案と効果（5分）：TOBE+ROI試算
4. 実施計画と承認事項（4分）：ロードマップ+リスク+承認フロー

## 4. 抵抗勢力マネジメント

**抵抗は「遅れて来る」。** 最初は賛成していた人が、具体化すると反対に変わる。これは失敗ではなく、プロジェクトが現実に近づいた証拠。

| ステークホルダー | 抵抗が顕在化するタイミング | 典型的な抵抗 | 対処 |
|---|---|---|---|
| **現場担当者** | TOBE設計確定時 | 「自分の仕事がなくなる」 | 再配置先を具体的に提示・スキルアップ研修を約束 |
| **人事部** | 人員削減計画確定時 | 「労務問題が起きる」 | 「削減」ではなく「再配置」。退職強要ではなく自然減・異動を設計 |
| **IT部門** | PoC設計時 | 「セキュリティ・保守負担が増える」 | 設計段階からIT部門を参加させ、共同設計者にする |
| **親会社物流ユニット** | 配車・物流変更時 | 「勝手にルールを変えるな」 | ドラフト段階で相談。「変更前に必ず報告」を約束 |

**語り方の鉄則：**
- NG：「7名削減します」→ 「誰が削られるのか」という恐怖を生む
- OK：「戦略業務に人を再配置します。単純作業はシステムが担います」→ 「自分は価値の高い仕事に移れる」という希望を提示

## 5. ROI試算フレーム

**基本計算式：**
```
ROI（%）= （削減効果合計 - 投資額合計） ÷ 投資額合計 × 100
回収期間（年）= 投資額合計 ÷ 年間削減効果
```

**太陽油脂型試算例：**

| 項目 | 試算値 |
|---|---|
| 削減人員 | 7名 |
| 平均人件費（社会保険含む） | 700万円/人/年 |
| **年間削減効果** | **4,900万円/年** |
| AI/RPAシステム開発費 | 2,000万円（一時費用） |
| 保守・運用費 | 300万円/年 |
| **総投資額（5年） ** | **3,500万円** |
| **5年間純利益** | **21,000万円** |
| **ROI（5年）** | **500%** |
| **回収期間** | **約0.5年** |

※ 試算は仮定値。実際はPoC後に精緻化する。役員会資料には「前提条件」を明示すること。

## 失敗パターンと回避策

| 失敗パターン | 発生場面 | 回避策 |
|---|---|---|
| PoC省略で本番導入 | スケジュール圧迫時 | M2でPoC前提の段階承認フローを明示 |
| 役員への過大約束 | プロジェクト受注時 | ROI試算に「前提条件」を必ず記載 |
| 人事との連携が遅れる | プロジェクト後半 | プロジェクト開始時から人事を巻き込む |
| 抵抗が顕在化してから対処 | 設計完了後のヒアリング | 設計段階から現場担当者を参加させる |
| 親会社の事後承認になる | 設計完了後に報告 | ドラフト段階での根回しをプロセスに組み込む |

## 想定質問と回答

**Q1：「役員会で『本当に7名削減できるのか』と問い詰められたら？」**

**質問の背景：** 役員（特にCFO・人事役員）は「人員削減の根拠の確かさ」を問う。楽観的な試算に基づく計画は信頼を失う。

**回答（推奨）：** 「7名削減はPoC結果を踏まえた段階承認型です」と明示する。PoC前の段階では「削減可能性：最大7名、最小5名」というレンジを提示し、PoC後に精緻化する設計を説明する。不確実性を隠す提案より、不確実性を管理する提案の方が役員の信頼を得やすい。また「削減」ではなく「再配置」として提示することで労務リスクを低減できることも付け加える。

他の選択肢との比較：「必ず7名削減できます」という断言は短期的に承認を得やすいが、失敗時の信頼失墜が大きい。段階承認型の方が長期的なパートナーシップ構築に有効。

---

**Q2：「現場担当者が強硬に反対している場合、どう対処するか？」**

**質問の背景：** 業務改革部が「現場の巻き込みがうまくいっていない」と悩む典型的な局面。プロジェクトリーダーとしての対処能力を問われる。

**回答（推奨）：** まず反対の「本音」を個別ヒアリングで聞く。表面上の反対（「システムが複雑すぎる」）の裏には本音（「自分の仕事がなくなるのが怖い」）が隠れていることが多い。本音が「雇用不安」なら再配置計画を具体化する。「業務理解の不足」なら設計の共同作業に参加させる。「過去の失敗体験」なら小さな成功体験を早期に作る（クイックウィン）。強硬な反対が続く場合は、その担当者の上長・業務改革部と連携してエスカレーションする。

## 用語集

| 用語 | 説明 |
|---|---|
| M2（マイルストーン2） | プロジェクトの第2段階完了を確認する検査ポイント |
| PoC（Proof of Concept） | 本番展開前に小規模で技術・業務可能性を検証する実証実験 |
| ROI（投資対効果） | 投資額に対してどれだけの利益が得られるかを示す指標 |
| クイックウィン | 短期間・低コストで実現できる小さな成功体験 |
| 段階承認フロー | フェーズごとに承認を取り直す意思決定設計 |
| スイムレーン図 | 担当者・組織ごとに業務フローを可視化した図 |
| 再配置 | 削減対象業務の担当者を別業務・部門に異動させること |
| エスカレーション | 問題を上位の意思決定者に引き上げて解決を求めること |
| KPIモニタリング | 設定した指標を定期的に測定・管理すること |
