## 第9章 AI 固有のインシデントレスポンス

### 9.1 AI 固有のインシデントタクソノミー

従来のインシデント（不正アクセス・マルウェア）に加え、AI システム特有のインシデントタイプが存在する。

| インシデントタイプ | 具体例 | 重大度目安 | 初期対応 |
|---|---|---|---|
| **プロンプトインジェクション** | システムプロンプト漏洩・認可迂回 | High | 該当ユーザーを封鎖・アクセスログ保全 |
| **機密情報漏洩** | LLM が PII を出力 | Critical | サービス停止・影響範囲特定・報告孉話 |
| **ハルシネーション大量発生** | 誤った回答が大量流通 | Medium | 該当デプロイメントを準正バージョンにロールバック |
| **モデル异常挙動** | 差別的・有害な回答 | High | モデル封ス・税理 AI 回答に切り替え |
| **AI エージェント超過行動** | 意図せずファイル削除・DB 書き改め | Critical | エージェント封鎖・影響範囲の調査・ロールバック |
| **モデルコスト異常** | DoS ・誤用によるコスト爆発 | Medium | コストアラート・レートリミット発動 |

### 9.2 インシデント対応フロー（AI 版）

```
[検知]
 監査アラート or ユーザー報告 or 正期スキャン
    ↓
[トリアージゥ]
 重大度判定（Critical/High/Medium/Low）
 影響範囲特定（テナント・API・モデル）
    ↓
[封じ込め]
 Critical/High → 即時サービス停止 or アクセス制限
 Medium/Low   → ユーザー封鎖 + ログ保全
    ↓
[調査]
 監査ログの全延 request_id でリクエスト履歴追跡
 RAG 場合: 取得ドキュメント ID の確認
 エージェント場合: ツール呼び出し履歴の確認
    ↓
[根本原因分析]
 OWASP LLM Top 10 のどのカテゴリか？
 システムプロンプトの設計不備か？アクセス制御の欠陥か？
    ↓
[復旧]
 設定変更またはパッチ適用
 モデルレスポンスの場合はデプロイメント切り替え
 動作テスト後にサービス再開
    ↓
[再発防止]
 セキュリティ対策を本フレームワークに反映
 活動結果をインシデントデータベースに登録
 四半期レビューで再発状況を確認
```

### 9.3 シナリオ別プレイブック

#### シナリオ A: システムプロンプト漏洩

```
検知トリガー:
  監査ログの出力に「You are a helpful assistant」等の
  システムプロンプトのキーワードが含まれる

即時対応（1時間以内）:
  1. 該当デプロイメントのアクセスを封鎖
  2. 漏洩テキスト範囲の特定（ログの request_id 連結）
  3. 影響ユーザー数の確認
  4. システムプロンプトを更新・再デプロイ

山留まり対応Ｈ36時間以内）:
  - 漏洩テキストが機密度「Internal」以上の場合は個人情報保護法第26条の漏洩報告を検討
  - 競合情報が含まれる場合はリーガル・ PR 対応を検討
```

#### シナリオ B: AI エージェントの超過行動

```
検知トリガー:
  監査ログに正規ホワイトリスト外の
  ツール呼び出し・ファイル操作・外部 API 呼び出し

即時対応（30分以内）:
  1. エージェントプロセスを即時停止
  2. 屖行済み操作の特定（ツール呼び出し履歴を全気通し確認）
  3. 影響を受けたリソースの特定・ロールバック
  4. プロンプト修正 + 最小権限設計を再検討
  5. Human-in-the-loop 承認ゲートを追加して再デプロイ
```

### 9.4 報告対象・タイミング

| 対象 | タイミング | 報告内容 |
|---|---|---|
| **社内 AI ガバナンス委員会** | 重大度 Medium 以上: 24時間以内 | インシデント様式・影響範囲・対応状況 |
| **CISO / CDO** | Critical: 即時， High: 2時間以内 | 事業影響・PR リスク・強化策 |
| **男座等 AI コンサルタント（委託先）** | 48時間以内 | 経緯・再発防止策・該当システムの修正内容 |
| **個人情報漏洩の場合** | 72時間以内（全訳法第26条） | 小委员会への報告必須 |

### 9.5 後日対応・学習ループ

| アクティビティ | タイミング | アウトプット |
|---|---|---|
| 根本原因分析（5 Whys） | 復旧後 1 週間以内 | 原因・再発防止策・完了期限 |
| セキュリティ対策の本フレームワークへの反映 | 2 週間以内 | 対策追加・ PR マージ |
| Red Team テストの再実施 | 1 か月以内 | 修正が有効か検証 |
| インシデントデータベース登録 | 即日 | 次回以降の検知筋度改善に活用 |
