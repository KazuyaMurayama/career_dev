# レポート4：物流・配車手配の自動化 ― 外部依存業務の自動化設計

---

## 1. 配車手配業務の構造

配車手配は「社内で完結しない」業務の代表例。自動化設計の前に業務構造を正確に把握することが必須。

**配車手配の5ステップ：**

| ステップ | 内容 | 主な担当 |
|---|---|---|
| ①発注情報確認 | 受注量・納品先・納品日時の確認 | 社内（受注担当→配車担当） |
| ②車両手配 | 必要台数・車格・ドライバーの選定 | 社内担当 or 物流会社 |
| ③ルート設定 | 納品先の順序・時間帯制約の最適化 | 社内担当 or 物流会社 |
| ④指示書発行 | ドライバーへの出発・積載・納品先指示 | 社内担当 |
| ⑤進捗追跡 | 配送中のステータス確認・遅延対応 | 社内担当 ↔ 物流会社 |

> **例え話：** 配車手配は「タクシー会社のディスパッチャー」の仕事に似ている。お客様（受注）の要望を聞き、ドライバーを割り当て、渋滞・遅延があれば調整する。ただし製造業では「積み荷の重量・温度管理・納品時間枠」など制約が多く、複雑度が格段に高い。

---

## 2. 自動化できる部分・できない部分

| 業務 | 自動化適性 | 理由 |
|---|---|---|
| 受注→配車要件変換 | 即時自動化 | ルール化可能（量×梱包→台数換算） |
| ルート最適化計算 | 即時自動化 | 最適化アルゴリズム（Google OR-Tools等） |
| 指示書PDF自動生成 | 即時自動化 | テンプレート＋データ差し込み |
| 物流会社への発注 | 条件付き自動化 | 物流会社のシステム連携が前提 |
| 遅延時の代替手配 | ハイブリッド | アラート自動発報 → 担当者が判断 |
| 突発大口注文の対応 | 人間担当 | 車両確保・コスト判断が必要 |
| 物流会社との価格交渉 | 人間担当 | 関係性・情報非対称が高い |

**重要な前提条件：**
物流会社側がデジタルデータ連携（API・EDI・メール自動処理）に対応していることが自動化の大前提。対応していない場合、「社内業務は自動化できるが物流会社への連絡は手動」という中途半端な状態になる。

## 3. 物流会社・ドライバーとのデータ連携設計

**連携方法の選択肢（現実的な順）：**

| 連携方式 | 難易度 | コスト | 向いているケース |
|---|---|---|---|
| メール添付（構造化CSV） | 低 | 低 | 物流会社がシステム未整備の場合 |
| EDI（電子データ交換） | 中 | 中 | 大手物流会社との取引 |
| API連携 | 高 | 高 | 物流会社が自社システムを持つ場合 |
| 物流プラットフォーム経由 | 中 | 中 | ハコベル・オープンロジ等のSaaS活用 |

**実務上のポイント：** 自動化の設計は「最も制約が多いレイヤー」に合わせる。物流会社がメールしか対応していないなら、まずRPAでメール自動生成＋送信を自動化し、「人がメールを書く」手作業を削る。

## 4. 親会社物流ユニットとの調整

太陽油脂のような「親会社物流ユニットが存在するケース」は、配車手配の権限が分散しており自動化設計が複雑になる。

| 課題 | 詳細 | 対処 |
|---|---|---|
| 配車ルール変更権限 | 親会社ユニットが配送基準・業者選定を管理 | 変更前に親会社ユニットの承認を取得する合意フローを設計 |
| コスト負担の所在 | 物流コストが親会社経由で計上される場合あり | 自動化投資のROI試算に親会社の承認が必要か事前確認 |
| データアクセス | 配送実績データが親会社システムに存在する場合 | データ共有の合意・APIアクセス権限を交渉 |
| 緊急時対応フロー | 遅延・事故時の連絡先・決裁ラインが不明確 | 例外処理フローを親会社ユニットと共同設計 |

**根回しのタイミング：** 自動化設計の「ドラフト段階」で相談する。完成後に持ち込むと「なぜ事前に言わなかったのか」という反発が起きる。

## 5. 太陽油脂型：配車手配自動化の段階設計

**現状（ASIS）想定：**
- 担当者がExcel・電話・メールで手動調整
- 受注確定後に翌日配車を手配するリードタイムがある
- 遅延情報は物流会社からの電話連絡が中心

**フェーズ1（PoC）：**
- 受注データ→配車要件自動変換（台数・車格・ルート初期提案）
- 指示書自動生成（Excel/PDFテンプレートへの自動差し込み）
- 担当者は確認・承認のみ

**フェーズ2（本番）：**
- 物流会社への発注メール自動送信（担当者のメール作業ゼロ化）
- 進捗追跡ダッシュボード自動更新（配送ステータス一覧）

**フェーズ3（発展）：**
- ルート最適化アルゴリズム導入（燃料コスト・時間削減）
- 遅延予測AIアラート（交通情報・天気データ連携）

**期待効果：**
- 配車手配工数：▲50〜60%
- 指示書作成エラー：ほぼゼロ化
- 担当者の再配置：遅延対応・物流会社交渉などの判断業務へ集中

## 6. 例外処理のハイブリッド設計

**例外発生パターンと対処設計：**

| 例外パターン | 自動対応（AI/RPA） | 人間対応 |
|---|---|---|
| 配送遅延（交通渋滞） | アラート自動発報・顧客通知ドラフト生成 | 担当者が顧客に連絡・追加対応判断 |
| 積み残し発生 | 翌日配送スロットの空き自動検索 | 担当者が顧客承諾・優先順位判断 |
| 緊急・当日追加注文 | 車両空き自動確認・料金試算 | 担当者がコスト判断・物流会社交渉 |
| 車両故障・事故 | 代替車両リスト自動提示 | 担当者が手配・顧客対応 |

**ハイブリッド設計の原則：** 「情報収集・選択肢提示はAI、最終判断と顧客対応は人間」。AIが情報整理の時間を削減することで、担当者は判断に集中できる。

## 失敗パターンと回避策

| 失敗パターン | 発生場面 | 回避策 |
|---|---|---|
| 物流会社連携を後回しにした | 社内自動化だけ先行した | 物流会社のシステム対応確認を設計フェーズ初期に実施 |
| 親会社承認を忘れた | 実装完了後に報告 | ドラフト段階で親会社ユニットに相談 |
| 例外処理フローが未設計 | 遅延発生時に対応が止まった | 例外パターンをヒアリングし、事前にフロー設計 |
| ルート最適化が現場に合わない | 現場制約（時間枠・車格）をシステムが把握していない | 現場担当者とルール設計のワークショップを実施 |

## 想定質問と回答

**Q1：「物流会社がシステム連携に対応していない場合はどうするか？」**

**質問の背景：** 地方の中堅物流会社を使っているケースでは、EDI・APIへの対応がないことが多い。顧客（業務改革部・物流担当）が現実的な代替案を求めている。

**回答（推奨）：** 3つの選択肢がある。①RPA＋メール自動化（物流会社側に変更不要・即時実施可）、②物流会社に対しシステム化投資を依頼する（費用分担交渉が必要）、③デジタル対応済みの物流会社への切り替えを検討する（中長期）。短期的には①から着手し、次契約更改時に②③を交渉テーブルに乗せるのが現実的。

他の選択肢との比較：物流会社変更は取引関係・コスト面でリスクが高い。まず①で効果を示してから交渉する方が社内説得コストが低い。

---

**Q2：「配車の自動化で事故が起きた場合の責任所在はどうなるか？」**

**質問の背景：** 法務・役員がシステム起因の事故責任を問う場面。特に自動化ルートが原因で遅配・事故が発生した場合のリスク管理を問われる。

**回答（推奨）：** システムが「提案」し人間が「承認・実行」する設計にすることで、最終決定者を人間に明確化する。ルート最適化の結果はドライバー・担当者が確認・承認するフローを必ず組み込む。また運送事業法上の責任はキャリア（物流会社）にあることを契約で明確化する。完全自動実行（人間承認なし）は現時点では法的リスクが高い。

## 用語集

| 用語 | 説明 |
|---|---|
| 配車手配 | 輸送に使う車両・ドライバーを割り当て、ルートと積載を指示する業務 |
| EDI（電子データ交換） | 企業間で標準フォーマットのデータを自動交換する仕組み |
| ルート最適化 | 複数の配送先を最短・最低コストで回る順序を計算するアルゴリズム |
| OR-Tools | Googleが提供するルート最適化等の組み合わせ最適化ライブラリ |
| ディスパッチャー | 車両・ドライバーの配車調整を担当する担当者 |
| 物流プラットフォーム | 荷主と物流会社をデジタルでマッチングするSaaSサービス |
| 車格 | 車両の大きさ・積載量の種別（4t車・10t車・軽バン等） |
| 積み残し | 注文量に対して配送便が不足し、当日配送できなかった荷物 |
